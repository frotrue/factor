## 1. 🔴 즉시 해야 할 작업

### 1.1 끊긴 고급 루프 복구

**현재 상태**: 게임의 후반부 성장 경로가 두 곳에서 막혀 있다. 이것이 해결되지 않으면 웨이브 10 이후의 진행감이 사실상 사라진다.

#### 문제 A: Fiber 케이블 해금 불가

| 항목 | 내용 |
|------|------|
| 원인 | `CONFIG.CABLES.FIBER`가 `TECH_FIBER_OPTIC`를 요구하나 `CONFIG.RESEARCH`에 해당 연구가 없음 |
| 위치 | `src/config.ts:260` |
| 영향 | 케이블 업그레이드 진행감 단절, 물류 병목 해소 수단 부재 |

**해결 방안**:
- `CONFIG.RESEARCH`에 `TECH_FIBER_OPTIC` 노드를 추가한다.
- 권장 위치: `TECH_DISTRIBUTED_AP` 이후 또는 독립 노드
- 권장 비용: 130~160 (Basic cable에서 한 단계 업그레이드 느낌)

```
연구 트리 예시:
TECH_DISTRIBUTED_AP
└── TECH_FIBER_OPTIC (Cost: 150)
    "광섬유 케이블을 해금합니다. 대역폭 8, 큐 20으로 대용량 데이터 전송이 가능합니다."
```

#### 문제 B: Inference Unit 생산 불가

| 항목 | 내용 |
|------|------|
| 원인 | `INFERENCE_UNIT_PRODUCTION` 레시피가 `ENERGY` 아이템을 요구하나, `ENERGY`를 아이템으로 생산하는 경로가 없음 |
| 위치 | `src/config.ts:291-295` |
| 영향 | Neural Trainer의 최종 레시피 사용 불가, 후반 방어 목표 단절 |

**해결 방안 (택 1)**:

| 방안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A: Energy Miner 확장 | Miner가 ENERGY 패치 위에서 `ENERGY` 아이템 생산 | 기존 건물 활용, 직관적 | Miner 로직 분기 추가 필요 |
| B: 레시피 수정 | `ENERGY` 대신 `SILICON` 또는 전력 소비로 대체 | 가장 빠른 수정 | 테마 일관성 약화 |
| C: 신규 건물 | 전력을 ENERGY 아이템으로 변환하는 "Energy Converter" 추가 | 물류 선택지 추가 | 작업량 가장 많음 |

**권장**: 방안 A가 가장 균형적이다. Miner가 자원 패치 타입에 따라 `SILICON` 또는 `ENERGY`를 생산하도록 수정하면 기존 ENERGY 패치 인프라를 그대로 활용할 수 있다.

- HUD/빌드바/액션바 겹침
- 모달(연구/설정/게임오버) 스크롤

### 2.2 저장/로드 회귀 안정화

**현재 위험**:
- `SaveData.version = '1.0.0'`이지만 버전별 마이그레이션 함수가 없음
- 신규 건물(Recycler, Data Cache), 신규 적(DDoS Bot), 난이도 설정이 추가되면서 저장 데이터 책임이 커짐
- 이전 버전 저장 파일 로드 시 호환성 미보장

**필수 작업**:

```
1. SaveData.version을 기준으로 최소 마이그레이션 함수 추가
   - v1.0.0 → v1.1.0: 누락 필드 기본값 채움
2. 저장/로드 회귀 테스트 체크리스트:
   - 신규 건물(Recycler, Data Cache) 배치 후 저장/로드
   - 케이블 큐에 아이템이 있는 상태 저장/로드
   - 난이도 설정 유지 확인
   - 웨이브 진행 중 저장/로드
   - DDoS 스웜 활성 상태 저장/로드
3. 저장 대상 vs 비저장 대상 문서화
```

### 2.3 Normal 20웨이브 밸런스 패스

**목표**: Normal 기준 20분 플레이에서 납득 가능한 난이도 곡선을 만든다.

**현재 웨이브 구성 수식** (`WaveManager.startWave()` 기준):

| 구간 | 적 수 공식 | HP 배율 |
|------|-----------|---------|
| 1~5 | 4 + wave | 1.0 |
| 6~15 | 8 + wave×1.5 | 1.25 + (wave-5)×0.04 |
| 16+ | 18 + wave×1.8 | 1.8 + (wave-15)×0.12 |

**검증이 필요한 항목**:

| 항목 | 질문 |
|------|------|
| 시작 자원 | Silicon 30개로 기본 생산+방어 조합을 만들 수 있는가? |
| 첫 실패 시점 | Normal에서 첫 실패가 웨이브 몇에서 오는가? |
| DDoS 체감 | 웨이브 8+ DDoS 스웜이 Filter 없이 버틸 수 있는가? |
| Recycler 효용 | 실제로 Recycler를 건설하는 것이 이득인가? |
| Data Cache 효용 | 케이블 병목이 체감되어 Data Cache가 필요한가? |
| 난이도 차이 | Easy/Hard/Nightmare가 수치 뻥튀기가 아닌 실질적 압박을 주는가? |

**권장 측정 방법**:
- `StatsManager` 초안을 먼저 만들어 웨이브별 Core HP, Silicon 잔량, 방어 타워 수, 적 처치 수를 기록한다.
- Normal 기준 10웨이브/20웨이브 도달 여부를 기준점으로 잡는다.

## 3. 🟢 중기 작업 (2~4주)

### 3.1 신규 전략 콘텐츠

QoL과 밸런스가 안정된 뒤 추가한다. 기존 `NEW_CONTENT_IDEAS_AND_PLAN.md`와 `PROJECT_ANALYSIS_AND_ROADMAP.md`의 추천을 종합하면 아래 순서가 최적이다.

| 순서 | 콘텐츠 | 이유 | 예상 작업량 |
|------|--------|------|------------|
| 1 | **EMP Tower** | DDoS/고속 적 대응 수단, 현재 군중제어 부재를 해결 | 중 |
| 2 | **Honeypot** | 방어 배치 전략 분기, Firewall과 역할 차별화 | 중 |
| 3 | **Ransomware 적** | 생산 라인 위협을 만들어 후반 긴장감 제공 | 상 |
| 4 | **Deepfake 적** | 위장 기믹으로 추가 전략적 판단 요구 | 상 |

**신규 건물 추가 시 수정 파일 체크리스트**:
1. `src/config.ts` — BUILDINGS 섹션 추가
2. `src/types.ts` — BuildingType 유니온 추가
3. `src/buildings/[Name].ts` — 건물 클래스 생성
4. `src/buildings/BuildingFactory.ts` — case 추가
5. `src/buildings/BaseBuilding.ts` — 시각적 외형 추가
6. `src/config.ts` → RESEARCH — 해금 연구 노드 추가
7. `src/managers/SaveManager.ts` — 저장/로드 호환성 확인

| 영역 | 완성도 | 비고 |
|------|--------|------|
| 핵심 생산 루프 | ★★★★☆ | RAW_DATA → WEIGHT_UPDATE 작동, 고급 루프 막힘 |

| 연구 시스템 | ★★★☆☆ | 구조 좋음, 일부 연구 누락 |

| 모바일 대응 | ★★☆☆☆ | 1차 구현됨, 재검증 필요 |
| 정보 표시 | ★★☆☆☆ | 기본 표시 있으나 병목 진단 부족 |
| 코드 구조 | ★★☆☆☆ | 핵심 파일 비대, `as any` 다수 |
| 테스트/QA | ★☆☆☆☆ | 자동 테스트 없음 |

### 핵심 장점

- **독창적 콘셉트**: AI 학습 파이프라인을 공장 자동화로 표현
- **유기적 시스템 연결**: 생산 → 방어 → 연구의 선순환
- **이중 물류**: 데이터(케이블) + 물리 자원(컨베이어) 분리
- **구역별 전력망**: 지역 블랙아웃이 전략적 의미를 가짐

### 핵심 위험

- **후반 루프 단절**: Fiber/Energy/Inference Unit 막힘으로 후반 성장감 부재
- **텍스트 깨짐**: 사용자 경험과 발표 품질을 크게 저하
- **밸런스 미검증**: 실제 플레이 데이터 없이 수식만으로 조정됨
- **코드 비대화**: 기능 추가 속도가 구조 정리 속도보다 빠름
