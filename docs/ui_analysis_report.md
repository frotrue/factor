> **분석일**: 2026-06-04  
> **대상 프로젝트**: Gradium (Phaser 3 + TypeScript + Vite, 2D 공장 자동화/타워 디펜스)  
> **분석 범위**: 전체 UI 레이어 (DOM HUD, Phaser 캔버스 메뉴, 모달, 모바일)

## 1. 현재 UI 아키텍처 개요

### 1.1 렌더링 이원화 구조

Gradium의 UI는 **두 가지 렌더링 레이어**가 혼재합니다:

| 레이어 | 기술 | 담당 화면 |
|--------|------|-----------|
| **Phaser Canvas** | Phaser Text/Graphics | 메인 메뉴, 난이도 선택, 게임 내 건물/적/지형 |
| **DOM Overlay** | HTML + CSS | HUD 상단바, 우측 전술 패널, 빌드 콘솔, 모달, 툴팁, 로그 |

> [!WARNING]
> **이원화 렌더링의 문제점**: 메인 메뉴(Phaser Text)와 인게임 UI(DOM)의 시각 언어가 완전히 단절됩니다. 전환 시 사용자는 전혀 다른 UI 체계를 경험하게 됩니다.

## 2. 디자인 시스템 평가

### 2.1 색상 팔레트

현재 CSS 변수:

```css
--neon-cyan: #4dd8ff;
--neon-magenta: #f06cff;
--neon-green: #64f58d;
--bg-dark: #05070c;
--state-warning: #f6c453;
--state-danger: #ff6676;
--state-success: #64f58d;
```

| 항목 | 평가 | 점수 |
|------|------|:----:|
| 색상 일관성 | 네온 사이버 테마 방향은 잡혀있으나 Phaser 메뉴와 DOM UI 간 불일치 | ⭐⭐⭐ |
| 대비 & 가독성 | 어두운 배경 위 밝은 텍스트, 대체로 양호 | ⭐⭐⭐⭐ |
| 의미적 색상 사용 | 성공/경고/위험 상태 색상이 체계적으로 정의됨 | ⭐⭐⭐⭐ |
| 캔버스-DOM 통일성 | `visualTheme.ts`와 CSS 변수가 분리되어 있고 값도 상이 | ⭐⭐ |

### 2.2 타이포그래피

- **단일 폰트**: `Share Tech Mono` (monospace)
- **평가**: 모노스페이스 폰트가 기술적 분위기를 살리지만, 모든 UI에 단일 폰트를 사용하면 **시각적 위계(hierarchy)가 약해짐**
- **제목/본문/캡션 구분**이 주로 `font-size`에만 의존하며, weight/family 변화가 없음
- 폰트 크기 범위: 8px ~ 64px (메뉴 타이틀), 대부분의 UI는 9~16px 범위에 집중

> [!NOTE]
> 모노스페이스 폰트는 숫자/데이터 표시에는 좋지만, 긴 설명문이나 UI 라벨에는 가독성이 떨어질 수 있습니다.

### 2.3 레이아웃 시스템

- CSS Grid와 Flexbox를 적절히 사용
- `clamp()`, `min()`, CSS 변수를 활용한 반응형 처리
- **하지만** 간격(spacing)과 크기(sizing)에 일관된 스케일 시스템이 없음
  - gap: 3px, 5px, 6px, 7px, 8px, 10px, 12px, 14px... 등 산발적
  - padding: 2px~24px 범위에서 비일관적

### 2.4 컴포넌트 디자인

| 컴포넌트 | 상태 | 주요 문제 |
|----------|------|-----------|
| Glass Panel | ✅ 글래스모피즘 적용 | backdrop-filter 의존 — 성능 비용 |
| 버튼 (build-btn) | ✅ hover/active 상태 있음 | 너무 많은 버튼 클래스가 하나의 규칙 블록 공유 |
| 탭 (tab-btn) | ✅ 네온 글로우 효과 | 선택 상태의 시각적 피드백이 너무 미묘 |
| 모달 | ⚠️ 기본적 | 열기/닫기 애니메이션 없음, 오버레이 배경 없음 |
| 툴팁 | ✅ 위치 자동 보정 | 디자인이 기본적, 화살표 없음 |
| 프로그레스 바 | ✅ 그라데이션 | 얇음(5px), 상태 변화 애니메이션 없음 |

- ❌ 배경 파티클이 단순하고 느림

## 4. 기술적 부채 분석

### 4.1 아키텍처 문제

| 문제 | 심각도 | 설명 |
|------|--------|------|
| **UIManager 모놀리스** | 🔴 높음 | 898줄의 단일 파일이 모든 HUD 로직을 중앙집중식으로 관리. `innerHTML` 직접 조작, DOM 질의, 이벤트 바인딩이 혼재 |
| **Phaser/DOM 이원화** | 🔴 높음 | 메인 메뉴는 Phaser, 인게임은 DOM. 전환 시 스타일 불연속, 유지보수 이중 비용 |
| **단일 CSS 파일** | 🟡 중간 | 1,280줄의 `main.css`에 모든 컴포넌트 스타일이 혼합. 네임스페이스 없는 클래스명 |
| **인라인 스타일 남용** | 🟡 중간 | `element.style.display`, `style.color` 등 JS 내 직접 스타일 조작이 빈번 |
| **ResearchUI stub** | 🟡 중간 | 28줄의 no-op — 연구 트리 UI가 완전히 비활성화된 상태 |
| **하드코딩된 DOM ID** | 🟡 중간 | 50개 이상의 `document.getElementById()` 호출이 문자열 ID에 의존 |

### 4.2 성능 우려

- `backdrop-filter: blur(14px)` — 모든 glass-panel에 적용, 모바일 성능 저하 가능
- innerHTML 재조립이 빈번 (빌드 버튼, Training Lab 전체 re-render)
- DOM 업데이트 throttling은 있으나 (250ms) 일부 경로는 매 프레임 실행

### 4.3 접근성

- ❌ ARIA 속성이 거의 없음 (일부 `aria-label`만 존재)
- ❌ 키보드 네비게이션 미지원 (탭 순서, 포커스 관리 부재)
- ❌ 색상만으로 상태 구분 — 색맹 사용자 고려 없음
- ❌ `font-size`가 8-9px인 요소 다수 — 최소 가독 크기 미달

## 6. 전면 재개편 시 권장 방향

### 6.1 아키텍처

1. **DOM 통합**: 메인 메뉴를 Phaser Text에서 DOM으로 전환하여 전체 UI의 일관성 확보
2. **컴포넌트 분리**: UIManager를 기능별 모듈로 분해 (HudBar, BuildConsole, TacticalRail, ModalSystem 등)
3. **CSS 모듈화**: `main.css`를 컴포넌트별로 분리하거나, CSS custom properties 기반 디자인 토큰 체계 구축
4. **상태-뷰 분리**: 현재의 직접 DOM 조작을 선언적 패턴으로 전환

### 6.2 비주얼

1. **타이포그래피 확장**: 제목용 sans-serif 폰트 추가 (예: Rajdhani, Orbitron), 데이터는 모노스페이스 유지
2. **아이콘 시스템**: 건물/자원/적에 SVG 아이콘 도입으로 식별성 향상
3. **마이크로 애니메이션**: 수치 변화, 모달 전환, 상태 변화에 시각 피드백 추가
4. **깊이감(depth)**: 다중 레이어 그림자, 미세한 그라데이션, 블러 레벨 차등으로 Z축 위계 표현

### 6.3 UX 개선

1. **빌드 시스템**: 건물 아이콘 개선, hover 프리뷰, 드래그 앤 드롭
2. **정보 계층화**: 필요한 정보만 보이도록 점진적 공개(progressive disclosure)
3. **모달 시스템**: 오버레이 딤, 열기/닫기 애니메이션, ESC 닫기 통합
4. **게임 오버**: 극적 연출, 통계 시각화, 메인메뉴 복귀 옵션

### 6.4 모바일

1. **터치 타겟 확대**: 최소 44×44px (Apple HIG 기준)
2. **바텀시트 패턴**: 정보 패널을 스와이프 가능한 바텀시트로
3. **제스처 통합**: 건물 배치/회전의 터치 최적화

> [!IMPORTANT]
> 전면 재개편 시 Playwright E2E 테스트(`tests/e2e/app-smoke.spec.ts`)가 DOM ID에 의존하고 있으므로, ID 변경 시 테스트도 함께 업데이트해야 합니다.

## 8. 추가 UI 상태 분석

### 8.1 튜토리얼 시스템

튜토리얼은 우측 레일에 `#tutorial-panel`로 표시되며:
- 타이프라이터 텍스트 애니메이션 (22ms/char + 사운드)
- 체크리스트 스텝 (완료=초록, 현재=시안, 비활성=회색)
- 인게임 가이드 하이라이트 (펄스 네온 박스, 흐름 화살표, 영역 원)
- 스킵 버튼

**문제**: 가이드 시각 요소는 Phaser Graphics인데, 스텝 설명은 DOM 패널이라 두 레이어의 시각적 연결이 약함.

### 8.2 웨이브 결과 알림

- 7초간 표시되는 글래스 카드 (화면 상단 중앙)
- 입장/퇴장 애니메이션 포함 (`waveResultIn`, `waveResultOut`)
- 2×2 그리드로 적 격파, 데이터 처리, 코어 무결성, 건물 피해 표시

**문제**: 일시적이라 놓칠 수 있고, 히스토리를 볼 수 있는 방법이 없음.

### 8.3 인게임 오버레이 (F1/F2)

- F1: 방어 범위 — 타워 주변 빨간 원
- F2: 전력망 — 노란 그리드 시각화
- 전력 경고: 무전력 건물에 펄스 노란 마커
- 버퍼 풀: 가득 찬 건물에 오렌지/파란 원

**문제**: 토글 키가 표시되지 않아 발견하기 어려움. 동시 활성화 시 시각적 충돌 가능.
